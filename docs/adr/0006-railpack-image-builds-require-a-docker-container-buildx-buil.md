# Railpack image builds require a docker-container buildx builder (overlay2 disables the mergeop the railpack frontend needs)

Live test building a real Astro repo proved that the railpack BuildKit frontend (invoked via docker buildx --build-arg BUILDKIT_SYNTAX=ghcr.io/railwayapp/railpack-frontend) uses BuildKit's mergeop LLB primitive, which Docker's DEFAULT overlay2 image store disables — the build fails with 'requested experimental feature mergeop has been disabled on the build server: only enabled with containerd image store backend'. This contradicts the earlier assumption that the buildx+BUILDKIT_SYNTAX path simply reuses the Docker daemon's built-in BuildKit with no extra setup. Decision: Fleet's railpack build path must run through a dedicated 'docker buildx create --driver docker-container' builder (BuildKit in its own container, full feature set incl. mergeop, result brought back with --load), OR require the operator to enable Docker's containerd image store. We prefer provisioning/using a docker-container builder because it works regardless of the operator's host image-store setting and needs no daemon reconfiguration. Note: railpack's native 'railpack build' instead wants a standalone buildkitd via BUILDKIT_HOST; we do not use that path. Verified end-to-end: railpack detected the Astro static site (Node 22, npm, dist/ + Caddy serve) with zero config, built a 159MB image via the docker-container builder, and the image served the site at HTTP 200.
